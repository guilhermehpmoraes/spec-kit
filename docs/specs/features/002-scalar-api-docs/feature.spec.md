## 1. Context

The Admin backend currently uses `@nestjs/swagger` to generate an OpenAPI specification and serves the default Swagger UI at `/api/docs`. While functional, Swagger UI has a dated appearance and limited developer experience (DX) compared to modern alternatives.

[Scalar](https://scalar.com/) is a modern, open-source API reference UI that renders OpenAPI specs with a significantly improved visual design, better navigation, built-in request testing, and enhanced readability. It integrates with NestJS via `@scalar/nestjs-api-reference`, consuming the same OpenAPI document generated by `@nestjs/swagger` — meaning all existing decorators (`ApiProperty`, `ApiOperation`, `ApiResponse`, `ApiBearerAuth`, `ApiTags`, etc.) remain unchanged.

This feature replaces only the documentation UI layer. The OpenAPI spec generation pipeline stays the same.

## 2. Scope

### In Scope

- Install `@scalar/nestjs-api-reference` as a dependency
- Replace the Swagger UI setup in `main.ts` with the Scalar API reference middleware
- Serve the Scalar documentation UI at the existing `/api/docs` route
- Configure Scalar with Bearer Auth as the preferred security scheme and persistent authentication (`persistAuth: true`) so developers authenticate once and the token persists across reloads
- Add an authentication guide description in the OpenAPI spec (`DocumentBuilder.setDescription`) that instructs developers how to obtain and use a JWT token directly from the docs
- Standardize all existing `@nestjs/swagger` decorators across the Admin backend to a consistent, comprehensive documentation pattern:
  - Every `@ApiProperty` must include `description` and `example`
  - Every non-void endpoint must have a response DTO class with `@ApiResponse({ type: ... })`
  - Create `LoginResponseDto` and `RefreshResponseDto` for auth endpoints (currently returning untyped objects)
  - Add swagger decorators to `AppController` (health check)
- Establish the decorator standard in the Admin architecture spec as the baseline for all future endpoints
- Establish Scalar as the standard API documentation UI for all NestJS backends in the monorepo
- Remove the Swagger UI setup code (no longer needed after Scalar replaces it)

### Out of Scope

- Custom Scalar theming, branding, or logo (default theme is sufficient for now)
- Exposing a separate raw OpenAPI JSON/YAML endpoint
- Removing `@nestjs/swagger` as a dependency (it is still required for OpenAPI spec generation)
- Frontend documentation or API client generation tooling
- Changing API behavior, routes, or business logic — only documentation metadata changes

## 3. User Scenarios (Prioritized)

### Scenario 1 - Developer browses API documentation (P1)

A developer navigates to `/api/docs` on the Admin backend and sees the Scalar API reference UI instead of the old Swagger UI. The documentation displays all existing endpoints with their parameters, request/response schemas, and authentication requirements, rendered in Scalar's modern interface.

#### Acceptance Scenarios

1. **Given** the Admin backend is running, **When** a developer navigates to `http://localhost:3000/api/docs`, **Then** the Scalar API reference UI is rendered with all documented endpoints visible.
1. **Given** the Admin backend is running, **When** a developer views an authenticated endpoint in Scalar, **Then** the Bearer Auth scheme is pre-selected and ready for token input.
3. **Given** the Admin backend is running, **When** a developer navigates to `http://localhost:3000/api/docs`, **Then** the old Swagger UI is no longer rendered.

### Scenario 2 - Developer authenticates once and tests multiple endpoints (P1)

A developer enters their JWT token once in the Scalar Bearer Auth input. The token is persisted in `localStorage` via Scalar's `persistAuth` option. Subsequent requests and page reloads retain the token, eliminating repeated authentication.

#### Acceptance Scenarios

1. **Given** the Scalar UI is loaded, **When** a developer enters a JWT token in the Bearer Auth field, **Then** the token is applied to all subsequent requests made from the Scalar UI.
2. **Given** a developer has previously entered a JWT token, **When** they reload the `/api/docs` page, **Then** the token is still present (persisted in localStorage).

### Scenario 3 - Developer tests API requests from documentation (P2)

A developer uses Scalar's built-in API client to send test requests directly from the documentation page without switching to an external tool like Postman or curl.

#### Acceptance Scenarios

1. **Given** the Scalar UI is loaded, **When** a developer selects an endpoint and fills in parameters, **Then** they can execute the request and see the response directly in the Scalar interface.

## 4. Functional Requirements

- **FR-001**: The system MUST serve the Scalar API reference UI at `/api/docs`.
- **FR-002**: The system MUST use the OpenAPI document generated by `@nestjs/swagger` as the data source for Scalar.
- **FR-003**: The system MUST NOT break any existing `@nestjs/swagger` decorator functionality.
- **FR-004**: The system MUST remove the legacy Swagger UI setup from `main.ts`.
- **FR-005**: The Scalar integration MUST be the standard approach for all NestJS backends in the monorepo.
- **FR-006**: The Scalar UI MUST have Bearer Auth pre-selected as the preferred security scheme.
- **FR-007**: The Scalar UI MUST persist authentication tokens in `localStorage` so developers authenticate only once across page reloads.
- **FR-008**: The OpenAPI spec description MUST include a step-by-step authentication guide visible at the top of the Scalar documentation.
- **FR-009**: Every `@ApiProperty` in existing DTOs MUST include `description` and `example` fields.
- **FR-010**: Every non-void controller endpoint MUST have a typed response DTO referenced in `@ApiResponse({ type: ... })`.
- **FR-011**: Auth login and refresh endpoints MUST have dedicated response DTO classes (`LoginResponseDto`, `RefreshResponseDto`).
- **FR-012**: The `AppController` health check endpoint MUST have `@ApiTags`, `@ApiOperation`, and `@ApiResponse` decorators.
- **FR-013**: The API documentation decorator standard MUST be documented in the Admin architecture spec.

## 5. Engineering Quality Constraints

- **EQ-001 (KISS)**: The implementation MUST prefer the simplest design that satisfies all acceptance scenarios — a minimal Scalar setup with default configuration.
- **EQ-002 (Self-descriptive code)**: New code MUST be understandable through naming and structure without relying on comments.
- **EQ-003 (Comments policy)**: Comments MUST be added only for non-obvious intent, tradeoffs, or constraints.
- **EQ-004 (Testability)**: The bootstrap change MUST be verifiable via an e2e test that confirms `/api/docs` responds with the Scalar UI.
- **EQ-005 (Patterns baseline)**: Design choices MUST follow current ADR baseline unless explicitly justified.

## 6. Dependencies and Risks

### Dependencies

- `@scalar/nestjs-api-reference` npm package
- `@nestjs/swagger` (existing — continues to generate the OpenAPI spec)

### Risks

- **Scalar NestJS integration compatibility**: The `@scalar/nestjs-api-reference` package must support the current NestJS version (v11). -> **Mitigation**: Verify compatibility before implementation by checking Scalar docs and NestJS version support.
- **Breaking change to existing docs URL**: Consumers bookmarking `/api/docs` should not be affected. -> **Mitigation**: Route remains the same (`/api/docs`).

## 7. Success Criteria

- **SC-001**: Navigating to `/api/docs` on any NestJS backend renders the Scalar API reference UI.
- **SC-002**: All existing API endpoints and their schemas are correctly displayed in Scalar with full request/response documentation.
- **SC-003**: Every DTO `@ApiProperty` has `description` and `example`. Every non-void endpoint has a typed `@ApiResponse`.
- **SC-004**: The old Swagger UI setup code is fully removed.
